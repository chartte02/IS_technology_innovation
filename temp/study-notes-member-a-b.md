## 开篇：从「PCAP 文件检测」说起

### 目前我们做的是什么？

**是的，NADS 目前的核心工作模式就是「PCAP 检测」。** 用最通俗的话解释：

> 把网络通信想象成邮政系统。每个 TCP 数据包 = 一封信。PCAP 文件 = 把某个时间段内所有经过网卡的信件全部拍成照片存下来的"监控录像带"。NADS 做的事情就是：**逐帧回放这卷录像带，检查每一封信里有没有恶意内容，发现异常就拉警报。**

```
PCAP 文件 (监控录像带)
  ↓ 逐帧回放
Scapy 拆包 (拆开信封)
  ↓ 读地址、读内容
ProtocolParser 解析 (翻译成人类能理解的格式)
  ↓
MisuseDetector (黑名单检查: 这封信里的内容像不像已知的攻击?)
  ↓
AnomalyDetector (行为检查: 这个发信人的行为是不是不正常?)
  ↓
AlertManager (统一拉警报)
  ↓
GUI 界面 (在屏幕上展示给安全运维人员看)
```

### 为什么从 PCAP 开始？

对初学者来说，PCAP（Packet Capture 的缩写）是最友好的起点：

| 模式 | 工作方式 | 难度 | 适合场景 |
|------|----------|------|----------|
| **PCAP 回放**（当前主模式） | 读取已保存的 `.pcap` 文件，逐包"播放"给检测引擎 | ⭐ 入门 | 学习、开发、测试、答辩演示 |
| **实时抓包** | 直接从网卡捕获经过的流量 | ⭐⭐⭐ 进阶 | 真实网络环境中部署 |
| **在线拦截（IPS）** | 不仅检测，还要在攻击到达目标前阻断 | ⭐⭐⭐⭐ 生产级 | 企业网络边界防护 |

**PCAP 回放的三大优势**：
1. **不需要管理员权限**：实时抓包需要 root/管理员运行，PCAP 回放不需要
2. **可重复测试**：同一份 PCAP 文件可以反复回放 100 次，每次结果一致，方便调 bug
3. **有标准数据集**：学术界公开了大量标注好的攻击 PCAP（如 CIC-IDS-2017），可以用来客观评估检测准确率

## 第一部分：成员A — 误用检测引擎 + 系统集成

### 1.1 已实现功能详解

#### 1.1.1 SignatureMatcher — 核心匹配引擎

**一句话概括**：把网络包里抓到的文字，跟一个"坏话词典"（YAML 特征库）快速对比，命中就报警。

**为什么需要这个模块？** 假设你有 250 条攻击规则，每条规则有一个关键词（比如 `UNION SELECT`、`<script>`、`../etc/passwd`），每个网络包 2000 字节。如果用最笨的方法——每条规则依次检查——那就是 250 × 2000 = 50 万次比较。每秒可能有几千个包，计算量很快就爆炸了。

`SignatureMatcher` 用了一种叫 **Aho-Corasick 自动机** 的算法（后面会详细讲），做到**不管有多少条规则，只扫描一次**就能找出所有匹配。

**核心设计：四层索引 + 三阶段匹配**

```
                    ┌──────────────────┐
                    │   match_packet   │
                    │   (收到一个包)     │
                    └────────┬─────────┘
                             │
            ┌────────────────┼────────────────┐
            ↓                ↓                ↓
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │ 端口索引      │ │ 协议索引      │ │ AC 自动机索引 │
    │ "目标端口80?" │ │ "是HTTP吗?"   │ │ "一次性扫描   │
    │ → 只查 Web   │ │ → 只查 HTTP  │ │  所有固定     │
    │   相关规则    │ │   相关规则    │ │  字符串模式"  │
    └──────────────┘ └──────────────┘ └──────────────┘
                             │
                             ↓ 候选规则列表 (已大幅缩减)
                    ┌──────────────────┐
                    │   三阶段匹配      │
                    │                  │
                    │  阶段1: AC扫描   │ ← O(n) 快速初筛
                    │  阶段2: 正则精匹  │ ← 处理变体模式
                    │  阶段3: 阈值判定  │ ← 统计累积触发
                    └──────────────────┘
```

**类比理解**：
- 端口索引 = 先看信封的收件窗口号。如果是 22 号窗口（SSH），就不会去查 SQL 注入规则（SQL 注入只发生在 HTTP 的 80 号窗口）
- AC 自动机 = 同时用 250 个荧光笔在信纸上划一遍，每个荧光笔只划自己关心的词，一遍划完就知道所有命中
- 正则精匹配 = AC 只能匹配"完全一样的字符串"，但攻击者可能写成 `UNION   SELECT`（多几个空格），正则能处理这种变体

#### 1.1.2 攻击特征库 — 7 个 YAML 文件，250+ 条规则

**一个特征库条目长什么样？** 以 SQL 注入检测为例：

```yaml
# signatures/sql_injection.yaml 中的一条规则
id: SQLI-001
name: "SQL Injection - UNION SELECT"
category: sql_injection          # 攻击类别
severity: critical               # 严重度：严重/高/中/低
description: "检测到 UNION SELECT 注入尝试"
patterns:                        # 匹配模式：发现这些文字就报警
  - "(?i)union\s+(all\s+)?select"         # (?i)=不区分大小写
  - "(?i)\bselect\b.*\bfrom\b.*\bwhere\b"
protocol: HTTP                   # 这条规则只检查 HTTP 流量
ports: [80, 443, 8080, 8443]    # 只检查这些端口的流量
```

**七类攻击覆盖**：

| 规则文件 | 检测什么 | 怎么检测（通俗版） |
|----------|----------|-------------------|
| `sql_injection.yaml` | SQL 注入 | 看请求里有没有 `UNION SELECT`、`' OR '1'='1`、`DROP TABLE` |
| `xss.yaml` | 跨站脚本 (XSS) | 看有没有 `<script>` 标签、`onerror=` 事件处理器 |
| `web_attack.yaml` | 目录遍历、命令注入 | 看有没有 `../../../etc/passwd`、`; whoami`、`| ls` |
| `brute_force.yaml` | 暴力破解 | 统计短时间内同一个 IP 的登录失败次数 |
| `backdoor.yaml` | 后门/木马 C2 | 看有没有 WebShell 特征（蚁剑/冰蝎）、挖矿协议 |
| `scan.yaml` | 端口扫描 / 漏洞扫描器 | 看 User-Agent 是不是已知扫描器（sqlmap/Nikto/Nmap） |
| `dos.yaml` | 拒绝服务攻击 | 统计 SYN 包占比是否异常高（> 80%） |

#### 1.1.3 IDSEngine — 把所有人写的模块组装起来

`main.py` 里的 `IDSEngine` 类就是"总指挥"——它不自己做检测，而是把一个包依次递给各个模块，把大家的输出汇总到 `AlertManager`。

```python
# IDSEngine._on_packet() 的核心逻辑（简化版）：
def _on_packet(self, packet):
    # 第1步：解析
    parsed = self.parser.parse(packet)        # 成员B: 拆信封
    
    # 第2步：误用检测（单包）
    alerts = self.misuse_detector.match_packet(parsed)  # 成员A: 黑名单
    
    # 第3步：TCP 流重组 → 流级匹配
    stream = self.reassembler.feed(parsed)    # 成员B: 拼图
    if stream:
        alerts += self.misuse_detector.match_stream(stream)  # 拼好的图再查一次
    
    # 第4步：异常检测
    self.anomaly_detector.update(parsed)       # 成员C: 更新统计
    
    # 第5步：基线学习
    if self.baseline_learner.is_learning():
        self.baseline_learner.feed(parsed)     # 成员C: 学习"正常"长什么样
    
    # 第6步：提交告警
    self.alert_mgr.submit_batch(alerts)        # 成员D: 统一输出
    
    # 第7步：每5秒检查一次异常
    if now - last_check >= 5:
        anomaly_alerts = self.anomaly_detector.check_all()
        self.alert_mgr.submit_batch(anomaly_alerts)
```

### 1.2 核心知识：Aho-Corasick 多模式匹配算法

#### 为什么这个算法这么重要？

> 如果你只能从 IDS 项目中学一个算法，就学 AC 自动机。

它是工业标准。Snort（全球最流行的开源 IDS）、Suricata、ClamAV（杀毒软件）的核心匹配引擎都基于它。答辩时讲清楚这个算法 = 直接证明你理解了 IDS 的核心。

#### 从零开始理解

**先回忆 KMP 算法（单模式匹配）**：
在 `"ABABABC"` 里找 `"ABABC"`，KMP 能做到 O(n)。秘诀是 `next` 数组：匹配失败时不是从头开始，而是跳到"当前已匹配部分的最长可匹配后缀"。

**AC 自动机 = KMP 的多模式版本**：
在 `"ushers"` 里同时找 `{he, she, his, hers}`，一次扫描全部找到。

**三个核心组件**：

```
1. Trie 树 (Goto 函数)
   把所有模式串建成一棵前缀树:
   
          root
         /    \
        h      s
       / \      \
      e   i      h
     /     \      \
    r       s      e
   /               ↑ 到这里匹配了 "he" + "she"
  s
  ↑ 到这里匹配了 "hers"
  
2. Failure Link (失配指针)
   匹配到 "she" 的 'e' 后，下一个字符是 'r' 但 "she" 没有 'r' 子节点
   → 不回到 root! 而是跳到 "he" 的 'e'（"she" 的最长可匹配后缀是 "he"）
   → 从那里继续检查有没有 'r' → 有! 走到 "her" 的 'r'

3. Output 继承 (最容易被忽略的细节!)
   匹配到 "she" → 不仅要输出 "she"，还要沿着 failure link 
   输出 "he"（因为 "he" 是 "she" 的后缀，也命中了一条规则）
```

**Python 实战**：
```python
import ahocorasick

# 1. 构建自动机：把所有攻击关键词加进去
A = ahocorasick.Automaton()
for rule_id, keyword in attack_keywords:
    A.add_word(keyword, (rule_id, keyword))
A.make_automaton()  # 构建 failure links

# 2. 扫描：payload 只遍历一次
for end_pos, (rule_id, keyword) in A.iter(payload):
    start_pos = end_pos - len(keyword) + 1
    print(f"发现攻击特征 '{keyword}' 在位置 {start_pos}，规则ID: {rule_id}")
```

## 第二部分：成员B — 数据采集层 + 协议解析 + 加密流量检测

### 2.1 已实现功能详解

#### 2.1.1 PacketCapture — "把网络包抓到手里"

**一句话概括**：像个渔网，按照你指定的条件（只捕 TCP 鱼、只捕 80 端口的鱼），把经过网卡的数据包捕上来。

**关键技术点**：

| 功能 | 怎么实现的 | 为什么要这样 |
|------|-----------|-------------|
| **异步抓包** | Scapy `AsyncSniffer`，在独立线程中运行 | 不阻塞 GUI 界面 |
| **BPF 过滤** | `filter="tcp port 80"` → 内核层面过滤 | 过滤发生在数据复制到程序之前，效率极高 |
| **PCAP 回放** | `rdpcap()` + 逐包 `_process_packet()` | 离线分析、重复测试、答辩演示 |
| **PPS 统计** | 滑动窗口计数（最近 N 秒的包数 / N） | 衡量抓包性能的核心指标 |

**BPF 过滤器速查**：
```
tcp                         → 只抓 TCP 包
tcp port 80                 → 只看 HTTP
tcp port 443                → 只看 HTTPS
host 192.168.1.100          → 只看某个 IP 的流量
tcp and (port 80 or port 443 or port 22)  → Web + SSH
```

#### 2.1.2 ProtocolParser — "拆开信封，翻译内容"

**一句话概括**：把原始的二进制网络数据翻译成人类可读的字典。

**三层解析**：

```
网络包 (一堆 bytes)
  ↓
┌─ IP 层 ─────────────────────────────────────┐
│  源IP: 10.0.0.5      从哪来的               │
```

#### 2.1.3 TCPStreamReassembler — "把撕碎的信拼回来"

**一句话概括**：攻击者可能把一个攻击拆成多个包发送来逃避检测，重组器把它们拼回完整内容。

**为什么需要它？**

```
攻击者把 "UNION SELECT" 拆成两个包：

包1: seq=100, payload="GET /?id=1 UNI"
包2: seq=118, payload="ON SELECT * FROM users--"

如果只看单个包 → 都看不出完整攻击特征
拼起来后 → "UNION SELECT" 完整出现 → 被 SQLI-001 规则命中！
```

**核心算法——严格序列号排序**：
```python
def add_segment(seq, payload):
    """
    seq=100, payload="ABC" → 期望下一个 seq=103
    
    三种情况:
    - seq == expected_seq → 正好, 直接拼上
    - seq <  expected_seq → 重传的旧数据, 忽略
    - seq >  expected_seq → 乱序了! 先暂存, 等中间的包到
    """
```

#### 2.1.4 TLSDetector — "不解密也能识破加密流量的伪装"

**一句话概括**：HTTPS 流量虽然内容加密了，但 TLS 握手阶段（建立加密连接时的"打招呼"）是明文的——从打招呼的方式就能识别出很多恶意软件。

**核心原理——JA3 指纹**：

```
客户端发送 ClientHello (明文):
  "你好! 我支持这些加密方式: [AES, DES, RC4, ...]
   我支持这些扩展功能: [...]
   我支持这些椭圆曲线: [...]"

JA3 指纹 = MD5(加密版本 + 加密套件列表 + 扩展列表 + 椭圆曲线列表)

不同的 TLS 库 (Chrome / Python / Cobalt Strike 恶意软件)
→ 不同的 ClientHello 参数组合
→ 不同的 JA3 指纹
→ 可以区分"正常流量"和"Cobalt Strike C2 通信"
```

**已知恶意 JA3 指纹示例**：
| 恶意软件 | JA3 指纹 | 说明 |
|----------|----------|------|
| Cobalt Strike | `37f463bf...` | 默认 TLS profile |
| Emotet | `51c64c77...` | 独特加密套件顺序 |
| Trickbot | `6734f374...` | 特定扩展组合 |
