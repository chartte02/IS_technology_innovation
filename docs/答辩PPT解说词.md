## 第 3 页 — 系统架构（60 秒）

> 这是系统的五层管道架构图。数据从最上层进入，经过五层逐层处理。

> **第一层数据采集层**，基于 Scapy 和 Npcap 实现实时抓包，支持 BPF 内核级过滤和 PCAP 文件回放。

> **第二层协议解析层**，识别 12 种应用层协议——包括 HTTP、HTTPS、SSH、FTP、SMTP、POP3、IMAP、DNS、TELNET、MySQL、TLS——并且实现了 TCP 严格序列号排序重组，防止攻击者通过 TCP 分片绕过检测。

> **第三层检测引擎层**是核心——左边是误用检测引擎，基于 AC 自动机多模式匹配，136 条规则；右边是异常检测引擎，7 种检测器加 Isolation Forest 无监督机器学习；还有攻击链关联分析器。

> **第四层告警管理层**，统一管理所有告警——去重、分级从 critical 到 low 四级、统计分析、JSON 导出。

> **第五层用户界面层**，基于 PyQt5 的 Apple 风格桌面应用，6 个功能页面，亮暗双主题。

> 右边列出了系统的关键数字——8 个核心模块、136 条规则、12 种协议、7 种检测器——这些数字后面会逐一展开。

> 这张图展示了一个数据包在系统中的完整旅行。

> 一个 PCAP 文件被读取后，首先经过 ProtocolParser 解析——提取 IP 地址、端口、TCP 标志位、HTTP 方法和 URI；然后同时送入误用检测和异常检测两条管线——误用检测用 AC 自动机和正则表达式做三阶段匹配，异常检测更新主机的 15 维统计画像并每 5 秒周期性检查；最后所有告警汇总到 AlertManager——去重、分级、推送给 GUI 实时显示。

## 第 5 页 — 误用检测引擎（90 秒）★ 核心页

> 误用检测引擎是整个系统的核心之一，基于 Aho-Corasick 自动机实现多模式匹配。

> AC 自动机本质上是一棵 Trie 前缀树加上 Failure Link 失配跳转。它把 136 条规则的所有固定字符串建为一棵树，共享公共前缀。匹配时沿着树走，失配了不回到根节点，而是跳到最长可匹配后缀状态——这是 KMP 算法的多模式推广。复杂度是 O(n+m)，n 是载荷长度，m 是模式总长度，与规则数量无关。

> 我们设计了三阶段匹配流水线：第一阶段 AC 自动机扫描 115 个固定字符串，一次遍历找出所有命中——95% 以上的正常流量在这一步就被过滤掉了。第二阶段对 AC 命中的候选规则执行 192 个预编译正则的精匹配。第三阶段对端口扫描、暴力破解这类需要统计累积的规则做阈值判定。

> 性能数据：正常包约 468 微秒，攻击包约 455 微秒。准确率 96%——SQL注入 80%、XSS 和 Web攻击 100%、暴力破解和后门 100%。误报率方面，自研的 78 条规则在正常流量中零误报。

> 我们还自研了 Suricata 规则导入器——可以解析业界标准的 Suricata 规则格式，自动转换成我们的 YAML 格式，使规则库可扩展到社区的三万多条规则。

## 第 6 页 — 加密流量检测（75 秒）★ PDF 必做①

> 加密流量检测是 PDF 要求的四项扩展功能之一。我们实现了 TLS JA3 和 JA4 指纹采集。

> JA3 的原理是：TLS 握手的 ClientHello 消息是明文发送的，包含五个字段——TLS 版本号、客户端支持的加密套件列表、TLS 扩展类型、支持的椭圆曲线和点格式。这五个字段拼接起来做 MD5 哈希，就得到了 32 字符的 JA3 指纹。

> 为什么 JA3 能识别恶意软件？因为不同的 TLS 库——Chrome 用的 Mozilla NSS、Python 用的 OpenSSL、Cobalt Strike 用的自定义 TLS 库——它们的 ClientHello 参数组合是不一样的。就像浏览器的 User-Agent 一样，JA3 指纹可以识别客户端软件。我们内置了已知恶意 JA3 指纹库，包括 Trickbot、Emotet 和 Cobalt Strike。

> 右边是我们实现的 C2 Beacon 心跳检测——这是第二个维度。恶意软件的 C2 通信有一个典型特征：固定间隔的心跳包，比如每 60 秒发一个包确认存活。我们通过变异系数 CV 来量化这个规律程度——CV 等于标准差除以均值。CV 越小越规律，CV 小于 0.05 几乎可以确定是 C2 流量。

> 加上证书异常检测——自签名证书、过期证书、弱加密套件——我们构成了三层防线。核心优势是：**完全不需要解密 HTTPS 载荷**，只靠明文握手信息和连接统计就能发现恶意 C2 通信。

## 第 7 页 — 异常检测 + ML（75 秒）★ PDF 必做②

> 异常检测引擎维护每台活跃主机的 15 维统计画像，包括连接数、SYN 数、访问的不同端口数和 IP 数、收发字节数、登录失败次数等。通过滑动时间窗口——60 秒窗口加 5 秒桶粒度——配合后台线程周期性检查。

> 我们实现了七种检测器：端口扫描检测同一个 IP 是否访问了超过 20 个不同端口；横向扫描检测是否访问超过 50 个不同目标 IP；SYN Flood 检测 SYN 包数量和占比；暴力破解检测登录失败累积次数；高频 DDoS 检测每秒包数；基线偏离检测当前指标是否远超历史基线。

> 右边是 ML 异常检测——PDF 要求的第二项扩展功能。我们选择了 Isolation Forest 无监督学习算法。它的核心思想很简单：异常数据"少且不同"，随机切几刀就能隔离出来；正常数据"多且相似"，需要很多刀才能隔离。每台主机映射到八维特征空间，用 Isolation Forest 预测——负一表示异常，正一表示正常。

> 我们还实现了两阶段精判：IF 粗筛标记异常主机后，进入三源投票——检查误用检测引擎是否有告警、基线 Profile 是否偏离超过两倍标准差、威胁情报是否标记为已知恶意。三票中至少两票确认为攻击，一票降级为低优先级，零票作为误报抑制。

> 参考了 sarthakghavghave 在 GitHub 上开源的两阶段方案，该方案报告了 88.6% 的误报消减率。我们在 10,000 包的稳定性测试中内存仅增长 2.9MB，处理速度超过 30 万包每秒。

## 第 8 页 — 攻击链 + 威胁情报 + 降噪（60 秒）★ PDF 必做③④

> 攻击链关联分析是 PDF 要求的第三项扩展。我们实现了 AttackChainAnalyzer——将同一来源 IP 在三十分钟时间窗口内的独立告警，按照 MITRE ATT&CK 模型串联成完整的攻击链路。七阶段映射覆盖侦察、利用、初始入侵、持久化、C2 通信、横向移动和最终影响。当同一个 IP 触发了三个以上不同阶段的告警时，系统输出攻击链告警。我们验证了四步攻击——扫描、暴力破解、SQL 注入、后门——被正确串联为侦察到持久化的完整攻击链。

> 威胁情报集成是我们的一大特色。我们接入了 AbuseIPDB——每天免费一千次查询——和 AlienVault OTX 两个平台，加上本地维护的恶意 IP 和 JA3 指纹黑名单。一旦确认来源 IP 已知恶意，告警自动升级 severity。

> 误报自动降噪是 PDF 第四项。我们实现了三级降噪体系：成员 A 负责的上下文感知过滤——根据攻击特征出现的位置来决定严重度，URL 参数中正常检测，Referer 和 Cookie 中出现的降级，白名单 IP 直接忽略。成员 C 负责的基线降噪——告警指标在正常基线两倍标准差范围内就降级。以及资产重要性配置——在 config.yaml 中定义四级资产，数据库服务器上的 SQL 注入告警升级为 critical，客户端子网的端口扫描降级为 low。

## 第 12 页 — 混合检测双引擎验证（60 秒）★ 答辩核心

> 这一页是我们混合检测架构的最直接验证。我们构造了一个 PCAP——同一个来源 IP 对目标发动了四阶段攻击：端口扫描 26 个端口、SQL 注入 5 种变体、SSH 暴力破解 8 次失败、后门 C2 连接。总共 55 个包，3 秒回放完毕。

> 检测结果：产生了 13 条去重告警。其中 1 条来自异常检测引擎——source 等于 anomaly，类型是 port_scan，因为 26 个端口偏离了正常基线。另外 12 条来自误用检测引擎——source 等于 misuse，包括 UNION SELECT、堆叠查询、报错注入、SQLMap 指纹识别等 8 种精确攻击特征。

> 右边是我们做的真实流量验证。Kali 扫描 Windows 的真实 PCAP——837 个包，识别出 12 种协议，双引擎同时告警。正常浏览流量——930 个真实 HTTP 请求——只产生 16 次误报，而且全部来自一条从 Suricata 社区导入的、匹配 curl 关键字的规则。自研的 78 条规则在正常流量中零误报。

> 这张图的结论：异常检测发现行为不正常，误用检测告诉你具体是什么攻击——同一次攻击事件，双引擎从不同维度同时告警，互相印证，互补短板。这就是混合检测架构的核心价值。
