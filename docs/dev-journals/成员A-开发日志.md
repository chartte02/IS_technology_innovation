### 2. 遇到的问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| AC 自动机中 `id(key)` 作字典 key 不可靠 | Python 的 `id()` 在对象回收后可被复用，导致 key 冲突 | 改用 tuple 直接作为 dict key（`self._ac_sig_map[key] = sig`），已由队友修复 |
| AC 模式未统一转小写 | 大小写不敏感匹配时，AC 自动机存入原始大小写的 pattern，无法匹配小写载荷 | `ac_pattern = pattern.lower()` 后再加入自动机，已由队友修复 |
| 测试文件 `test_signature_match.py` 被意外删除 | git 操作中丢失 | `git restore` 恢复 |
| Scapy 提示 "No libpcap provider available" | Windows 未安装 Npcap | 安装 Npcap（WinPcap 兼容模式），不影响测试运行 |

### 2. 遇到的问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| AC 自动机在 25 个模式下比暴力遍历慢 | 少量模式时 AC 自动机的 trie 遍历开销 > Python `in` 操作符的 C 实现 | 这是预期行为——AC 自动机在 100+ 模式时才有明显优势；当前 25 个字符串模式 + 261 个正则模式的混合架构是合理的 |
| 端口索引对 HTTP(80端口) 加速不明显 | 80 端口关联了 996 个索引条目（几乎所有 Web 规则），候选集等同于全量 | 确认索引设计正确——对罕见端口(9999)加速 10.5x，这正是索引的价值所在 |

### 3. Agent 协作记录

| 任务 | 是否用 Agent | 效果评估 |

| 设计三组性能对比实验 | ✅ 用了 | 自动设计 AC vs 暴力、索引 vs 全量、AC vs 遍历三次实验 |
| 数据解读与答辩素材提炼 | ✅ 用了 | 从实验数据中提取出"端口索引 10.5x 加速"作为答辩核心数据 |

### 4. 技术决策

| 决策 | 选项 A | 选项 B | 选择 | 理由 |
|------|--------|--------|------|------|
| 答辩性能数据展示重点 | AC 自动机加速 | 端口索引加速 | 端口索引加速 | AC 对 25 个模式效果有限，端口索引对非标准端口有 10.5x 提升，更有说服力 |
