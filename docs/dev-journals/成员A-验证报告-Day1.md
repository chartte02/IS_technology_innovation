## 2. 发现的问题

### 问题1：42个简单字符串模式静默丢弃 → 已修复

**根因**: `_build_indices()` 中 AC 自动机不可用时缺少 else fallback，导致纯字符串模式既不在 AC 也不在 regex，完全丢失。

**受影响的规则**: BACKDOOR-001/002/003、BRUTE-001/002/005 等的全部关键字

**修复**: 增加 `else` 分支，用 `re.escape(compiled) + re.IGNORECASE` 将简单字符串安全地转为正则匹配。

**代码位置**: `_build_indices()` L193-198

### 问题2：`(?i)`被误判为正则元字符 → 已修复

**根因**: `_classify_pattern()` 把 `(?i)` 列入正则元字符列表，导致200+个仅含大小写标志的模式走了 O(k×n) 的正则遍历匹配。

**修复**: 剥离内联标志后再判断；`flags` 作为参数传给 `re.compile()`。

**效果**: 正则匹配器从 265→192（减少 28%）

**代码位置**: `_classify_pattern()` L213-258

### 问题3：fallback 路径中 `(?i)` 被 `re.escape()` 变成字面量 → 已修复

**根因**: `_classify_pattern()` 返回的 `compiled = pattern.lower()` 仍包含 `(?i)` 前缀，导致 fallback 路径中 `re.escape("(?i)%3cscript%3e")` 把 flag 转义成了字面量 `\(\?i\)%3cscript%3e`。

**证明**: XSS-007（检测 URL 编码的 `<script>` 标签）修复前完全无法匹配任何 payload。

**修复**: `compiled = stripped.lower()` 使用已剥离内联标志的文本。

**代码位置**: `_classify_pattern()` L250-254 + `_build_indices()` L196

## 6. 已知限制

1. **SQLI-003 `(?i)--\\s`** 要求在 `--` 后跟空白符，但 MySQL 中 `--` 后跟任何非字母数字字符都是注释。`admin'--&pass=123` 未被检测。建议将模式放宽为 `(?i)--[^a-zA-Z0-9]` 或在 YAML 中增加变体。

2. **pyahocorasick 未安装**导致所有模式走正则遍历，性能比 AC 模式慢 20-40 倍。建议 `pip install pyahocorasick`。

3. **perf_profiler 覆盖率 34%**（32/93 规则可触发），因为仅用第一个 pattern 的脱敏文本无法匹配所有规则。建议扩展 perf_profiler 为各规则生成专属触发 payload。
