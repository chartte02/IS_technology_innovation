# 项目总体设计报告

**项目名称**：常见网络攻击检测系统 (Network Attack Detection System, NADS)

---

## 1、系统需求分析

### 1.1 需求分析

随着互联网攻击手段日益复杂化和多样化，传统防火墙和单纯的特征匹配检测已难以应对现代网络威胁。本系统针对以下安全问题进行增强：

- **Web 应用层攻击泛滥**：SQL 注入、XSS 跨站脚本、命令注入、文件包含、SSRF、XXE、SSTI 等攻击持续威胁 Web 服务安全；
- **暴力破解与凭证猜测**：攻击者通过 SSH、FTP、HTTP、RDP、Telnet 等协议进行大规模暴力登录尝试；
- **后门与木马通信隐蔽性强**：C2 命令与控制通信、WebShell 后门、挖矿木马、DNS 隧道等常绕过传统边界防护；
- **DoS/DDoS 拒绝服务攻击**：SYN Flood、Slowloris、HTTP Flood、DNS 放大攻击导致服务不可用；
- **扫描与探测行为**：Nmap、Nikto、SQLMap、Burp Suite 等扫描器的指纹识别和敏感路径探测为后续攻击铺路；
- **未知攻击与零日威胁**：单纯依赖特征签名无法检测未见过的攻击变种；
- **加密流量中的恶意行为**：TLS/SSL 加密通道可能隐藏恶意 C2 通信，传统 DPI 无法解析加密载荷；
- **告警疲劳问题**：大量重复告警和误报告警淹没真正的安全事件。

基于上述安全需求，系统采用 **误用检测 + 异常检测** 的混合架构，在特征匹配基础上引入统计基线偏离检测和机器学习方法，实现对已知攻击的高精度识别和未知行为的异常感知能力。

### 1.2 总体功能要求

本系统实现以下核心功能：

1. **实时网络流量采集**：基于 WinPcap/LibPcap 的底层抓包，支持 BPF 过滤和 PCAP 文件回放；
2. **多协议深度解析**：支持 TCP/IP 协议栈解析，识别 HTTP/HTTPS/SSH/DNS/FTP/SMTP/Telnet/MySQL/TLS 九种应用层协议，并提取 HTTP 方法、URI、Host、User-Agent 等字段；
3. **TCP 流重组**：对 TCP 分片包进行按序重组，防止攻击者通过分片逃避检测；
4. **误用检测（签名匹配）**：基于 Aho-Corasick 多模式匹配算法和正则表达式，加载 250+ 条 YAML 格式的攻击特征规则，覆盖 SQL 注入、XSS、Web 攻击、暴力破解、后门木马、DoS、扫描探测七大类；
5. **异常检测（基线偏离）**：每台主机维护多维度统计画像（连接数、SYN 数、唯一端口数、唯一 IP 数、字节数、登录失败次数等），通过滑动时间窗口检测端口扫描、横向扩散、SYN Flood、暴力破解、高频 DDoS 流量、基线偏离等异常行为；
6. **基线学习**：在正常网络环境中采集流量基线（流量分布、TOP 端口、HTTP 占比、服务器角色识别），支持基线保存/加载持久化；
7. **告警管理**：统一告警格式，支持去重（时间窗口内重复告警合并）、分级（critical/high/medium/low）、筛选查询、统计报表、JSON 文件导出；
8. **扩展功能——加密流量检测**：从 TLS ClientHello 提取 JA3/JA4 指纹，检测自签名证书、弱加密套件、异常 TLS 握手行为，识别恶意软件 C2 通信；
9. **扩展功能——ML 未知攻击检测**：基于 Isolation Forest 无监督机器学习模型，检测偏离正常行为模式的未知攻击和零日攻击；
10. **扩展功能——攻击链关联分析**：将单条独立告警串联为完整攻击链路（扫描→漏洞利用→外联 C2→横向扩散）；
11. **扩展功能——误报自动降噪**：结合基线数据、资产重要性配置和上下文行为分析，自动过滤和降级低质量告警；
12. **友好的桌面 GUI**：基于 PyQt5 的暗色主题图形界面，包含仪表盘、告警列表、统计分析图表、特征库管理、系统日志五大功能面板。

系统由四名成员分工协作开发，具体分工如下：

**成员A（项目负责人）—— 误用检测引擎 + 系统集成**

- 设计并实现基于 Aho-Corasick 多模式匹配算法的误用检测引擎 `SignatureMatcher`，包括 AC 自动机索引构建、正则预编译、端口和协议倒排索引，实现三阶段匹配流水线（AC 自动机快速初筛 → 正则精匹配 → 阈值型规则判定）
- 编写和维护 7 个 YAML 攻击特征库文件（SQL 注入、XSS、Web 攻击、暴力破解、后门木马、扫描探测、DoS/DDoS），累计 250+ 条检测规则，每条规则包含签名 ID、严重度分级、匹配模式和触发条件
- 设计并实现 `IDSEngine` 集成引擎，组装所有模块（数据采集→协议解析→流重组→误用检测→异常检测→告警管理）的回调链，确保数据包从采集到告警的全链路贯通
- 实现 `main.py` 主入口，支持 GUI 模式、命令行模式、PCAP 回放模式、基线学习模式四种运行方式
- 实现特征库热更新机制（`reload()`），支持不重启系统的规则动态加载
- 拓展功能：配合成员C 完成误报降噪中的上下文感知过滤（识别攻击特征出现位置——URL 参数 vs Referer vs Cookie vs Body，非关键位置匹配自动降级 severity）；扩展特征库规则数量；进行 AC 自动机 vs 暴力匹配的性能对比测试

**成员B —— 数据采集层 + 协议解析 + 加密流量检测**

- 设计并实现 `PacketCapture` 数据包捕获模块，基于 Scapy AsyncSniffer 实现异步非阻塞抓包，支持 BPF 过滤规则、混杂模式、环形缓冲区、PCAP 文件变速回放、网卡枚举和运行时统计（PPS/包数/字节数/运行时长）
- 设计并实现 `ProtocolParser` 协议解析模块，解析 IP/TCP 包头（IP 地址、端口、序列号、确认号、TCP 标志位），通过端口匹配和协议指纹识别九种应用层协议（HTTP/HTTPS/SSH/DNS/FTP/SMTP/Telnet/MySQL/TLS），对 HTTP 流量进行深度解析（method、URI、Host、User-Agent），输出标准化的 `parsed_packet` 字典
- 设计并实现 `TCPStreamReassembler` TCP 流重组模块，维护每个 TCP 连接的独立 `StreamBuffer`，支持严格序列号排序、重传包去重、乱序包缓存和按序拼接，采用 LRU 淘汰策略 + 后台定时清理防止内存泄漏，线程安全设计
- 拓展功能（PDF 必做）：新增 TLS 加密流量检测模块，从 TLS ClientHello 握手消息中提取 JA3/JA4 指纹（MD5 哈希：SSLVersion + CipherSuites + Extensions + EllipticCurves），检测自签名证书、过期证书、弱加密套件（RC4/DES/3DES/NULL）、非常规 TLS 版本协商，加载已知恶意 TLS 指纹库（Trickbot/Emotet/CobaltStrike）进行黑名单匹配
- 拓展选做：扩展更多应用层协议识别（POP3/IMAP/Redis/MongoDB）；进行不同 PPS 速率下的抓包丢包率性能测试；Windows/Linux 双平台适配测试并记录差异

**成员C —— 异常检测引擎 + 基线学习 + ML 检测**

- 设计并实现 `AnomalyDetector` 异常检测引擎，为每台活跃主机维护 `HostStats` 统计画像（15 个维度：连接数、SYN 数、唯一目标端口数、唯一目标 IP 数、收发字节数、包数、登录失败次数、首次/末次活跃时间等），通过滑动时间窗口（60 秒窗口 + 5 秒桶粒度）配合后台定时重置线程实现数据过期清理
- 实现七种异常检测器：端口扫描检测（单 IP 访问多端口）、横向扫描检测（单 IP 访问多目标）、SYN Flood 检测（SYN 包数量+占比）、暴力破解检测（登录失败累积次数）、高频流量检测（PPS 超阈值）、DDoS 检测（多源 IP 高频并发）、基线偏离检测（当前指标 vs 基线 μ±3σ），每种检测器独立阈值可配置
- 设计并实现 `BaselineLearner` 基线学习器，在正常流量期间采集多维度统计指标（按小时流量分布、TOP 端口排名、HTTP/HTTPS 协议占比、服务器角色识别），计算各指标的均值 μ 和标准差 σ，支持基线数据的 JSON 持久化保存和加载
- 拓展功能（PDF 必做）：新增 `MLAnomalyDetector` 基于 Isolation Forest 的无监督机器学习异常检测模型，从 HostStats 中提取 8 维特征向量（连接数/SYN 数/唯一端口数/唯一 IP 数/字节收发/包速率/登录失败次数），训练后自动识别偏离正常行为模式的未知攻击和零日攻击，替代固定阈值方案的局限性
- 拓展功能（PDF 必做）：实现攻击链关联分析，将时间窗口内同一来源 IP 的多条独立告警按攻击阶段串联为完整攻击链路（侦察扫描→初始访问/暴力破解→执行/漏洞利用→持久化/C2 通信→横向扩散），输出关联后的攻击链报告
- 拓展功能（PDF 必做）：配合成员A 完成误报自动降噪，负责基线降噪部分（当前告警指标在基线正常范围内则自动降级）、资产重要性配置（在 config.yaml 中定义服务器/客户端子网的重要性等级，低重要性资产的告警自动降级），输出降噪前后告警数量对比数据

**成员D —— GUI 界面 + 告警管理 + 测试 + 文档**

- 设计并实现 `AlertManager` 告警管理器，定义统一的 `Alert` 数据结构，实现告警去重（时间窗口内相同源/目标/类型的重复告警自动合并）、分级管理（critical/high/medium/low）、多维筛选查询（按严重度/类别/时间范围/IP）、统计报表生成（按时段严重度分布/类别分布/TOP 攻击来源/实时 60 秒统计）、JSON 文件持久化导出和文件日志记录，通过回调机制向 GUI 实时推送新告警
- 设计并实现 `IDSMainWindow` 主窗口界面（PyQt5），采用暗色主题风格，包含五个功能 Tab 页：仪表盘（5 个统计卡片 + 最近告警表 + 实时流量面板）、告警列表（表格展示 + 多维筛选 + 告警确认/误报标记 + 批量导出）、统计分析（PyQtChart 实现的告警严重度饼图、攻击类别柱状图、实时 PPS 流量折线图、TOP 攻击来源排行柱状图）、特征库管理（规则浏览/启用禁用）、系统日志（实时日志流 + 级别筛选）
- 实现 `DetectionWorker(QThread)` 后台检测线程，确保长时间抓包和检测运算不阻塞 GUI 响应，通过 Qt 信号槽机制实现后台与界面的线程安全通信；顶部控制栏实现启动/暂停、网卡切换、BPF 过滤规则输入、PCAP 回放文件选择；底部状态栏实时显示 PPS/告警数/运行时间
- 编写测试套件：`test_signature_match.py` 覆盖七大类攻击特征匹配的测试用例，验证检测准确率；为异常检测模块编写端口扫描/暴力破解/DoS 的模拟测试；端到端集成测试（PCAP 回放 → 检测 → 告警验证）；性能基准测试（不同 PPS 下的 CPU/内存占用）
- 搭建演示环境：准备至少 5 个含不同攻击类型的 PCAP 测试样本文件；搭建 VirtualBox 虚拟机演示环境（Kali 攻击机 + Metasploitable 靶机 + Host-Only 网络）；编写一键启动脚本和操作文档
- 负责答辩 PPT 制作：汇总各成员的技术贡献、性能数据和架构图，设计演示脚本和现场演示流程，准备现场网络故障时的 PCAP 回放备用方案

## 2、计划进度安排

本项目开发分为四个阶段，各阶段按模块并行推进、逐步集成：

**阶段一：环境搭建与模块独立跑通**

各成员完成开发环境配置，确保自己负责的模块能独立运行并输出正确结果。

- 全员：安装 Python 3.9+、配置 venv 虚拟环境、安装项目依赖（Scapy/PyYAML/PyQt5/PyQtChart/pyahocorasick），Windows 用户额外安装 Npcap 并验证抓包权限
- 成员A：修通 `SignatureMatcher` 的导入路径，从 YAML 特征库加载全部 250+ 条规则并验证按类别分布正确，构造测试用例验证三阶段匹配流水线（AC 自动机→正则→阈值）的基本功能
- 成员B：验证 `PacketCapture` 能在本机成功抓取 TCP 数据包，验证 `ProtocolParser` 能正确解析构造的 HTTP/TCP 包并输出标准 `parsed_packet` 字典，验证 `TCPStreamReassembler` 能对分片包完成流重组
- 成员C：阅读 `HostStats` 和 `AnomalyDetector` 数据结构，构造模拟数据验证七种检测器（端口扫描/横向扫描/SYN Flood/暴力破解/高频流量/基线偏离/DDoS）可独立触发告警，验证 `BaselineLearner` 的采集→计算→保存→加载全流程
- 成员D：验证 PyQt5 + PyQtChart 环境正常，验证 `AlertManager` 的去重、统计、JSON 导出功能，验证 GUI 主窗口能独立打开且五个 Tab 页正常切换

**阶段二：模块两两联调与拓展功能启动**

相邻模块之间进行接口联调，确保数据在不同成员负责的模块间正确传递。

- 成员A ↔ 成员B 联调（误用检测链路）：成员B 的 `ProtocolParser` 输出标准 `parsed_packet` → 成员A 的 `SignatureMatcher.match_packet()` 接收并正确匹配，验证单包检测链路的字段兼容性
- 成员B ↔ 成员C 联调（异常检测链路）：成员B 的输出 → 成员C 的 `AnomalyDetector.update()` 接收并正确更新主机统计画像，验证异常检测链路的字段兼容性
- 成员A（集成）：修通 `main.py` 中 `IDSEngine._on_packet()` 的完整回调链，确保数据从抓包→解析→流重组→误用检测→异常检测→告警管理全链路不报错
- 成员C（拓展启动）：开始 `MLAnomalyDetector`（Isolation Forest）的特征工程设计和模型训练代码
- 成员D（拓展启动）：完成 PyQtChart 四个图表（严重度饼图/类别柱状图/实时流量折线图/TOP 来源柱状图）的初始实现并接入 `AlertManager` 的真实数据

**阶段三：全员集成与拓展功能开发**

所有模块在 `IDSEngine` 中组装完毕，端到端全链路跑通，四位成员同步推进各自的 PDF 必做拓展功能。

- 全员联调：使用 PCAP 回放模式进行端到端测试（PCAP → 抓包回放 → 协议解析 → TCP 重组 → 误用检测 + 异常检测 → 告警管理 → GUI 实时显示），验证全链路数据流通和接口一致性
- 成员A（拓展）：上下文感知过滤降噪（识别攻击特征出现位置并分级处理）、特征库规则扩展（新增 15+ 条规则，覆盖 Webshell 蚁剑/冰蝎/哥斯拉特征）、AC 自动机 vs 暴力匹配性能对比数据采集
- 成员B（拓展 — PDF 必做）：TLS 加密流量检测模块开发，实现 JA3/JA4 指纹提取、证书异常检测（自签名/过期/弱加密套件）、已知恶意 TLS 指纹黑名单加载
- 成员C（拓展 — PDF 必做）：ML 模型（Isolation Forest）集成测试与参数调优、攻击链关联分析（扫描→漏洞利用→C2→横向扩散串联）、基线降噪 + 资产重要性配置（config.yaml 资产定义 → 告警自动降级）
- 成员D（拓展）：演示环境搭建（虚拟机方案：Kali + Metasploitable + Host-Only 网络；或 PCAP 方案：收集 5 个不同类型攻击的 PCAP 样本）、测试套件完善（异常检测模拟测试 + 端到端集成测试 + 性能基准测试）、答辩 PPT 初稿

**阶段四：测试验证与答辩准备**

全面测试系统稳定性，采集性能数据，完成答辩材料。

- 全链路稳定性测试：长时间运行测试（>1 小时），监控内存占用、告警去重准确率、GUI 响应延迟；PCAP 批量回放测试（多个攻击样本连续回放，验证检测覆盖率和告警完整性）
- 性能数据采集：特征匹配延迟（< 1ms/包，10000 包批量测试）、抓包 PPS（> 5000 pps，PCAP 高速回放）、GUI 刷新延迟（< 100ms，1 秒定时器）、内存占用（< 500MB，长时间运行监控）、ML 模型 vs 固定阈值方案对比（Precision/Recall/F1）

#### 各层功能说明

**第1层 — 数据采集层**（成员B）

| 模块 | 功能 |
|------|------|
| `PacketCapture` | Scapy AsyncSniffer 异步抓包，支持 BPF 过滤、混杂模式、PCAP 文件回放、网卡枚举、运行时统计（PPS/包数/运行时间） |

**第2层 — 协议解析层**（成员B）

| 模块 | 功能 |
|------|------|
| `ProtocolParser` | IP/TCP 头解析，9 种应用层协议识别（端口+指纹），HTTP 深度解析（method/URI/Host/UA），输出标准 `parsed_packet` 字典 |
| `TCPStreamReassembler` | 基于流的 TCP 包重组，LRU 淘汰 + 后台定时清理，线程安全，输出完整 TCP 流数据供检测引擎使用 |
| `TLSDetector`（拓展） | JA3/JA4 指纹采集，自签名证书/弱加密套件/异常握手检测，已知恶意 TLS 指纹黑名单匹配 |

**第3层 — 检测引擎层**（成员A + 成员C）

| 模块 | 功能 |
|------|------|
| `SignatureMatcher` | 加载 YAML 特征库，构建 Aho-Corasick 自动机 + 正则索引 + 端口/协议索引，三阶段匹配（AC→正则→阈值），支持流级别匹配和热更新 |
| `AnomalyDetector` | 每主机维护 15 维统计画像，7 种检测器（端口扫描/横向扫描/SYN Flood/暴力破解/高频流量/基线偏离/DDoS），滑动时间窗口 + 后台重置线程 |

| 攻击链关联（拓展） | 将独立告警按时间/IP 关联为攻击链路（侦察→初始访问→执行→持久化→C2） |

## 4、系统功能模块1：误用检测引擎（成员A）

### 4.1 模块结构

本模块负责基于已知攻击特征的签名匹配检测，是系统的核心检测引擎之一。

```
                  ┌─────────────────────────────────┐
                  │        main.py (IDSEngine)       │
                  │       集成引擎，组装调用          │
                  └─────────────┬───────────────────┘
                                │
                  ┌─────────────▼───────────────────┐
                  │       SignatureMatcher           │
                  │      （核心匹配引擎）              │
                  ├─────────────────────────────────┤
                  │  ┌───────────────────────────┐  │
                  │  │   _build_indices()        │  │
                  │  │   索引构建器               │  │
                  │  │   · AC 自动机构建          │  │
                  │  │   · 正则预编译             │  │
                  │  │   · 端口索引               │  │
                  │  │   · 协议索引               │  │
                  │  └───────────────────────────┘  │
                  │                                 │
                  │  ┌───────────────────────────┐  │
                  │  │   match_packet(parsed)     │  │
                  │  │   单包匹配（三阶段）         │  │
                  │  │   1. AC 自动机快速扫描      │  │
                  │  │   2. 正则精匹配             │  │
                  │  │   3. 阈值型规则判定         │  │
                  │  └───────────────────────────┘  │
                  │                                 │
                  │  ┌───────────────────────────┐  │
                  │  │   match_stream(data, info) │  │
                  │  │   流级别匹配（抗分片）       │  │
                  │  │   对 TCP 重组后的全流做匹配  │  │
                  │  └───────────────────────────┘  │
                  │                                 │
                  │  ┌───────────────────────────┐  │
                  │  │   reload()                 │  │
                  │  │   特征库热更新              │  │
                  │  └───────────────────────────┘  │
                  └─────────────────────────────────┘
                                │
                  ┌─────────────▼───────────────────┐
                  │     signatures/*.yaml (7 文件)   │
                  │     攻击特征规则库 (250+ 条)       │
                  ├─────────────────────────────────┤
                  │  sql_injection.yaml  (10 条)    │
                  │  xss.yaml            (10 条)    │
                  │  web_attack.yaml     (9 条)     │
                  │  brute_force.yaml    (6 条)     │
                  │  backdoor.yaml       (9 条)     │
                  │  scan.yaml           (9 条)     │
                  │  dos.yaml            (6 条)     │
                  └─────────────────────────────────┘
```

**子模块间协调关系**：
- `load_all()` 调用 `_build_indices()` 将所有规则加载到内存索引中；
- `match_packet()` 按优先级执行三阶段匹配：AC 自动机负责高速初筛（O(n+m) 复杂度），正则表达式负责精匹配，阈值型规则按端口/协议索引触发；
- TCP 流重组后的数据通过 `match_stream()` 进行全流级别的特征匹配，避免攻击者通过 TCP 分片绕过检测；
- `reload()` 支持不重启系统的情况下重新加载 YAML 特征库。

### 4.2 子模块1：Signature 特征数据结构

**功能**：定义单条攻击特征的完整数据结构。

**结构**：
```python
class Signature:
    id: str           # 规则唯一标识，如 "SQLI-001"
    name: str         # 规则名称，如 "SQL Injection - UNION SELECT"
    category: str     # 攻击大类，如 "sql_injection"
    type: str         # 具体类型
    severity: str     # 严重度: critical / high / medium / low
    description: str  # 人类可读描述
    patterns: list    # 匹配模式列表
    threshold: dict   # 阈值型规则的触发条件
    protocol: str     # 适用的应用层协议
    ports: list       # 适用端口列表
```

**技术实现**：从 YAML 文件反序列化，支持模式字符串中的正则标志、阈值条件等字段。

### 4.3 子模块2：SignatureMatcher 匹配引擎

**功能**：核心匹配引擎，负责加载特征库并执行多阶段匹配。

**技术实现**：

1. **AC 自动机加速**：使用 `pyahocorasick` 库构建 Aho-Corasick 自动机，将所有固定字符串模式一次性加载，单次遍历即可找到所有匹配，复杂度 O(n+m)（n 为载荷长度，m 为模式数量）。若 pyahocorasick 不可用，自动回退到 `SimpleMultiPatternMatcher` 纯 Python 实现。

2. **三阶段匹配流水线**：
   - **阶段1**（AC 自动机扫描）：对 payload 做一次遍历，找出所有命中模式的字节偏移位置；
   - **阶段2**（正则精匹配）：对阶段1 命中的候选，用预编译的正则表达式做精确匹配，提取匹配文本；
   - **阶段3**（阈值型规则）：对端口扫描、DDoS 等需要统计阈值判断的规则，匹配后检查累积计数是否达到阈值。

3. **端口与协议索引**：在 `_build_indices()` 中按端口和协议建立倒排索引，匹配时先查索引筛选相关规则，减少无效匹配次数。

4. **上下文感知降噪（拓展）**：在 `match_packet()` 中识别攻击特征出现的位置（URL 参数 vs Referer vs Cookie vs Body），对非关键位置的匹配自动降级 severity。

**逻辑处理**：
```
match_packet(parsed):
  1. 提取 payload
  2. 通过端口/协议索引筛选候选规则
  3. 对 payload 运行 AC 自动机 → 获取命中模式列表
  4. 对每个命中模式执行正则精匹配 → 提取 matched_text
  5. 检查阈值型规则的累积条件
  6. 组装 alert dict 列表返回
```

## 5、系统功能模块2：异常检测引擎（成员C）

### 5.1 模块结构

本模块负责基于统计行为的异常检测，通过学习正常网络基线来识别偏离行为。

```
                  ┌─────────────────────────────────┐
                  │        main.py (IDSEngine)       │
                  └─────────────┬───────────────────┘
                                │
        ┌───────────────────────┼───────────────────────┐

  ├──────────────┤    ├──────────────────┤    ├──────────────────┤
  │· 流量分布    │    │  HostStats       │    │  IsolationForest │
  │· TOP 端口    │    │  BaselineProfile │    │  8维特征向量     │
  │· HTTP 占比   │    │                  │    │  · 连接数        │
  │· 服务角色    │    │  7种检测器:      │    │  · SYN数         │
  │· 保存/加载   │    │  · 端口扫描      │    │  · 唯一端口数    │
  └──────────────┘    │  · 横向扫描      │    │  · 唯一IP数      │
                      │  · SYN Flood     │    │  · 字节收/发     │
                      │  · 暴力破解      │    │  · 包速率        │
                      │  · 高频流量      │    │  · 登录失败次数  │
                      │  · 基线偏离      │    └──────────────────┘
                      │  · DDoS          │
                      │                  │
                      │  辅助机制:       │
                      │  · 滑动时间窗口  │
                      │  · 后台重置线程  │
                      │  · 白名单机制    │
                      └──────────────────┘
```

**子模块间协调关系**：
- `BaselineLearner` 在正常流量期间学习基线，产出 `BaselineProfile` 供 `AnomalyDetector` 引用；
- `AnomalyDetector.update()` 每个数据包到达时更新对应主机的 `HostStats`；
- `AnomalyDetector.check_all()` 周期性（每5秒）对所有主机遍历执行 7 种检测器；
- `MLAnomalyDetector`（拓展）从 `HostStats` 中提取 8 维特征向量，用 IsolationForest 做无监督异常检测；
- 攻击链关联模块（拓展）从 `AlertManager` 获取历史告警，按时间和 IP 相关度串联攻击链路。

### 5.2 子模块1：BaselineLearner 基线学习器

**功能**：在受控的正常网络环境中采集流量数据，计算并持久化正常行为基线。

**技术实现**：
- `start_learning(duration)` 启动基线采集周期；
- `feed(parsed)` 接收每个已解析的数据包，按主机 IP 分组累积统计指标（连接数、字节数、端口分布、协议分布、时间分布）；
- `stop_learning()` 结束采集，调用 `_compute_baseline()` 计算统计量（均值 μ、标准差 σ、TOP 端口、HTTP/HTTPS 占比、流量按小时分布）；
- `save_baseline(path)` / `load_baseline(path)` 以 JSON 格式持久化基线数据。

**逻辑处理**：
```
BaselineLearner 工作流程:
  1. start_learning() → 标记 is_learning=True, 记录起始时间
  2. 每个包 → feed(parsed) → 按 src_ip 分组累加统计
  3. stop_learning() → _compute_baseline() → 输出 NetworkBaseline
  4. NetworkBaseline 包含:
     - learning_duration: 学习时长
     - sample_count: 总样本数
     - host_profiles: 每台主机的 HostBaseline
     - global_stats: 全局统计（总流量/平均 PPS/协议分布）
```

### 5.3 子模块2：AnomalyDetector 异常检测器

**功能**：实时监控每台主机的网络行为，通过 7 种检测器识别异常。

**技术实现**：

1. **HostStats 主机画像**（15 个统计指标）：
   ```
   conn_count, syn_count, unique_dst_ports, unique_dst_ips,
   bytes_sent, bytes_received, packet_count, login_failures,
   first_seen, last_seen, ...
   ```

2. **7 种检测器**：

| 检测器 | 检测原理 | 默认阈值 |
|--------|----------|----------|
| `_check_port_scan()` | 60 秒内访问不同端口数超过阈值 | > 20 端口 |
| `_check_horizontal_scan()` | 60 秒内访问不同 IP 数超过阈值 | > 50 IP |
| `_check_syn_flood()` | SYN 包数量超过阈值且 SYN 占比过高 | > 1000 且 > 80% |
| `_check_brute_force()` | 60 秒内登录失败次数超过阈值 | > 5 次 |
| `_check_high_frequency()` | 每秒包数超过阈值 | > 10000 pps |
| `_check_baseline_deviation()` | 当前指标偏离基线 μ ± 3σ | 可配置 |
| `_check_ddos()` | 多源 IP 高频同时发包 | PPS > 阈值 |

3. **滑动时间窗口**：60 秒窗口 + 5 秒桶粒度，支持数据过期自动清理。

4. **动态阈值自适应（拓展）**：在基线期计算每个指标的均值 μ 和标准差 σ，检测时若指标 > μ + 3σ 则告警，避免不同网络环境下手动调参。

5. **白名单机制**：支持配置忽略特定 IP/IP 段的检测。

## 8、系统出错处理设计

### 8.1 错误类型与处理方案

| 错误类别 | 产生原因 | 提示信息 | 处理办法 |
|----------|----------|----------|----------|
| **抓包权限不足** | 未以管理员身份运行 / Npcap 未安装 | `[ERROR] 无法打开网络接口: 权限不足。请以管理员身份运行或安装 Npcap` | 1. Windows: 以管理员运行终端 + 安装 Npcap（勾选 WinPcap 兼容模式）2. Linux: `sudo python main.py` |
| **网卡接口不存在** | 指定的接口名错误或网卡被禁用 | `[ERROR] 接口 '{name}' 不存在或不可用。可用接口: {list}` | 调用 `PacketCapture.list_interfaces()` 列出可用接口，重新指定 |
| **YAML 特征库解析失败** | YAML 文件缩进错误（混用 Tab 和空格）/ 格式不合法 | `[ERROR] 加载特征库失败: {filename} - {error_detail}` | 检查 YAML 文件缩进（必须用空格），验证 YAML 语法 |
| **pyahocorasick 不可用** | C 扩展库安装失败（编译环境缺失） | `[WARNING] pyahocorasick 未安装，使用纯 Python fallback（性能较低）` | 自动回退到 `SimpleMultiPatternMatcher`，功能正常但速度较慢。建议 `pip install pyahocorasick` |
| **PyQt5/PyQtChart 未安装** | 依赖未安装 | `[ERROR] 无法启动 GUI: {error}。请安装: pip install pyqt5 pyqtchart` | 安装缺失依赖，或使用命令行模式 `python main.py --console` |
| **配置文件缺失** | `config.yaml` 文件被删除或路径错误 | `[WARNING] 配置文件不存在: {path}, 使用默认配置` | 系统使用内置默认值继续运行，建议从模板恢复配置文件 |
| **Scapy 导入失败** | 依赖未安装 | `[ERROR] 导入失败: scapy。请运行: pip install -r requirements.txt` | 安装项目依赖 |
| **PCAP 文件不存在** | 回放路径错误 | `[ERROR] PCAP 文件不存在: {path}` | 检查文件路径，确保 PCAP 文件在指定位置 |
| **TCP 流重组内存溢出** | 并发流数超过 `max_concurrent_streams` | `[DEBUG] LRU 淘汰流: {key}` | 自动 LRU 淘汰最久未活跃的流，调整 `max_concurrent_streams` 配置 |
| **告警去重窗口内重复告警** | 同一攻击产生多条相同告警 | 静默处理（无用户可见提示，仅在 debug 日志记录） | 自动合并，返回 `None` 表示重复告警已过滤 |
| **JSON 导出文件写入失败** | 磁盘空间不足 / 权限不足 | `[ERROR] 无法写入告警文件: {path} - {error}` | 检查磁盘空间和目录写入权限 |
| **ML 模型未训练即调用 predict** | 基线学习未完成 | `[WARNING] ML 模型未训练，跳过预测` | 返回默认值（正常），等待 `train()` 被调用 |
| **模块间接口字段缺失** | 上下游模块版本不一致 | `[ERROR] KeyError: 缺少必要字段 '{field}' in parsed/alert dict` | 检查模块版本一致性，确保接口格式符合 `CLAUDE.md` 定义 |

### 8.2 系统容错设计原则

1. **优雅降级**：当可选组件不可用时（如 pyahocorasick、PyQtChart），自动回退到替代方案，核心检测功能不受影响；
2. **防御性编程**：所有模块对外部输入做字段校验，缺失字段给出明确错误信息而非静默崩溃；
3. **日志分级记录**：DEBUG（详细调试）、INFO（正常运行）、WARNING（可恢复异常）、ERROR（需要干预）；
4. **异常隔离**：单个模块的异常不影响其他模块运行。`_on_packet()` 回调中捕获所有异常，单个包的解析失败不影响后续包的处理；
5. **资源自动回收**：TCP 流通过 LRU 淘汰防止内存泄漏，告警内存有上限（默认 10000 条），后台线程支持 `shutdown()` 安全退出；
6. **线程安全**：涉及多线程访问的数据结构（`TCPStreamReassembler` 的流表、`AlertManager` 的告警列表）使用 `threading.RLock()` 保护。
