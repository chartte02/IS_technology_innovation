# 混合检测双引擎展示 — demo_hybrid_detection.pcap

> **文件**: `tests/test_pcaps/demo_hybrid_detection.pcap`
> **用途**: 答辩时展示"误用检测 + 异常检测"双引擎同时工作的核心优势
> **生成方式**: Scapy 构造，同一来源IP的四阶段攻击 + 正常流量混合

---

## 一、PCAP 内容

```
攻击者: 10.0.7.77 (Kali)
靶机:   192.168.1.100 (内网服务器)
总包数: 55 个

攻击阶段:
  Phase 1 (包1-26): 端口扫描 — 25个不同端口的SYN探测
  Phase 2 (包27-31): SQL注入 — 5种注入变体 (UNION SELECT / OR 1=1 / DROP TABLE / 报错注入 / SQLMap UA)
  Phase 3 (包32-39): SSH暴力破解 — 8次登录失败
  Phase 4 (包40): 后门C2连接 — AntSword UA 后门访问

正常流量: 15个HTTP请求 (来自 192.168.1.5~192.168.1.9)
```

## 二、预期检出结果

### 异常检测引擎 (source=anomaly)

通过**统计偏离**触发：

| 告警类型 | 检测依据 | 严重度 |
|----------|----------|:--:|
| port_scan | 正常用户访问5个以内端口，此IP访问了**26个不同端口** (阈值20) | medium |

> "异常检测不知道 Nmap 长什么样——它只看到这个 IP 的行为与基线严重偏离"

### 误用检测引擎 (source=misuse)

通过**特征匹配**触发：

| 规则ID | 告警名称 | 匹配模式 | 严重度 |
|--------|----------|----------|:--:|
| SQLI-001 | UNION SELECT注入 | `(?i)union.*select` | critical |
| SQLI-002 | Tautology 1=1 | `(?i)or\s+['"]?1['"]?\s*=\s*['"]?1` | high |
| SQLI-004 | DROP TABLE | `(?i)drop\s+table` | critical |
| SQLI-008 | 堆叠查询 | `(?i);\s*(select|insert|update|delete|drop)` | critical |
| SQLI-010 | SQLMap UA | `(?i)sqlmap/[0-9.]+` | medium |
| SQLI-011 | MySQL报错注入 | `(?i)extractvalue` | high |
| SUR-1000001 | Suricata UNION SELECT | content=`UNION SELECT` | high |
| SUR-3000002 | Suricata SQLMap | classtype=web-attack | high |

> "误用检测不知道什么算'正常'——它只是在流量中精确匹配了8条已知攻击特征"

## 三、GUI 演示操作

```
1. 回放 PCAP
   [📂 Replay PCAP] → 选择 demo_hybrid_detection.pcap

2. 切换到 ⚠ Alerts Tab
   → 查看告警表格的 "Source" 列
   → 看到两种来源: misuse 和 anomaly

3. 手指向关键行，讲解:
   ┌──────────┬──────────────┬─────────────────────────────┐
   │ Source   │ Category     │ 含义                         │
   ├──────────┼──────────────┼─────────────────────────────┤
   │ anomaly  │ scan         │ 异常检测: 统计偏离 → 端口扫描  │
   │ misuse   │ sql_injection│ 误用检测: 特征匹配 → 精确攻击名│
   └──────────┴──────────────┴─────────────────────────────┘

4. 双击一条 misuse 告警 → 弹窗展示匹配到的原文
   双击 anomaly 告警 → 弹窗展示统计详情 (26端口/阈值20)

5. 切换到 ◈ Attack Chain Tab → 展示同一IP的四步攻击链
   蓝色节点(10.0.7.77) → 黄(扫描) → 橙(SQL注入) → 橙(暴力破解) → 红(后门)
```

## 四、答辩解说词（可直接念）

> "接下来验证混合检测架构的核心优势。这个 PCAP 模拟了一次完整的网络攻击——同一个来源 IP 对目标发动了端口扫描、SQL 注入、暴力破解、后门连接四阶段攻击。
>
> 请大家注意告警表格的 Source 列——
>
> 这条 source=anomaly 的告警，是**异常检测引擎**通过统计偏离发现的：正常主机访问端口数在 5 个以内，这个 IP 短时间内访问了 26 个不同端口，严重偏离基线。
>
> 这 12 条 source=misuse 的告警，是**误用检测引擎**通过特征匹配发现的：UNION SELECT 对应 SQLI-001 规则，堆叠查询对应 SQLI-008 规则，SQLMap UA 对应 SQLI-010 规则。
>
> **同一次攻击，双引擎从不同维度同时产生了告警**——
> 异常检测发现'行为不正常'，误用检测告诉你'具体是什么攻击'。
> 两套告警来自不同维度，互相印证，互补短板。
> 这就是混合检测架构的设计理念。"

## 五、与 kali_to_windows_scan.pcap 对比

| 维度 | kali_to_windows_scan | demo_hybrid_detection |
|------|:--:|:--:|
| 包数 | 837 | **55** |
| 回放时长 | ~60秒 | **~3秒** |
| 攻击类型 | 1种 (仅扫描) | **4种** (扫描/SQLi/暴力破解/后门) |
| misuse告警数 | 5 | **12** |
| anomaly告警数 | 2 | **1** |
| 攻击链 | 1步 (仅侦察) | **4步** (侦察→利用→初始入侵→持久化) |
| 演示效果 | 等待时间太长 | **即回即看，Source列对比极其直观** |

**结论**: `demo_hybrid_detection.pcap` 是混合检测双引擎展示的**最佳选择**。`kali_to_windows_scan.pcap` 保留为备用——可用来证明系统能处理真实网络流量（非人工构造）。
